# Mufeng Brainstorm

## Overview

结构化头脑风暴框架。先发散（越多越奇越好）→ 再收敛（价值/成本矩阵）→ 深挖 Top 3 → 子 Agent 反驳 Top 1。

**四类场景：产品功能 / 技术选型 / 内容策略 / 项目规划**

## When to Use

- 项目初期，不知从哪里入手
- 功能设计有多个可能方向，需要梳理
- 技术选型举棋不定
- 内容创作选题困难
- 任何"我们可以怎么做"类问题

**不适用于：** 已经有明确方向只需执行的任务

## Process

### Step 0：识别场景

根据用户描述，判断属于哪类：

| 场景 | 关键词 | 跳转 |
|------|--------|------|
| 产品功能 | 新功能、用户需求、产品设计 | Phase A |
| 技术选型 | 架构、框架、迁移、性能 | Phase B |
| 内容策略 | 文章、选题、公众号、传播 | Phase C |
| 项目规划 | 新项目、方向、市场、调研 | Phase D |

如果无法判断，直接问用户。

---

### Phase A：产品功能头脑风暴

**发散（Step 1）** — 列出 20 个方向，不做评判：
- 5 个常规思路（用户最容易想到的）
- 5 个竞品已有方案（分析优劣）
- 5 个反常识路径（反向、小众、跨界）
- 5 个"资源无限"方向（理想态）

**收敛（Step 2）** — 4 象限矩阵：

```
           高价值
    规划长期 │ 优先执行
  ──────────┼──────────
    直接砍掉 │ 试水验证
           低价值
      高成本   低成本
```

**深挖（Step 3）** — Top 3 各自回答：
1. 核心假设 + 最快验证方式
2. 最大风险点
3. 2 周内可执行的 MVP

**反驳（Step 4）** — 用子 Agent 站在反对者立场，给出 Top 1 方向最有力的 3 个反对理由 + 应对策略

---

### Phase B：技术选型分析

**摸底（Step 1）** — 明确约束条件（如未提及则说明假设）：
- 团队技术背景、规模
- 预期用户量 / 并发规模
- 交付时间限制
- 长期维护预期

**候选（Step 2）** — 列出 5-8 个方案，每个包含：技术栈组成、适合场景、成功案例

**对比（Step 3）** — 表格对比：开发效率 / 性能上限 / 学习曲线 / 社区生态 / 维护成本 / 与现有栈兼容性

**推荐（Step 4）** — 明确首选 + 备选 + 切换信号 + 未来 1 年重新评估的触发条件

**风险（Step 5）** — 最容易被忽视的 3 个坑 + 规避方式

---

### Phase C：内容策略头脑风暴

**受众（Step 1）** — 定义 3 类读者：核心 / 潜在 / 意外

**选题发散（Step 2）** — 15 个方向：
- 5 个硬核干货（深度、专业）
- 5 个共鸣情感（故事、经历）
- 5 个热点借势（时效、话题）

**筛选（Step 3）** — Top 3 评分维度：与作者身份匹配度 / 差异化程度 / 传播潜力

**框架（Step 4）** — Top 1 的：
- 标题 3 个变体（不同角度）
- 一句话摘要
- 开头钩子 → 核心内容 → 行动号召
- 可引用数据 / 案例 2-3 个

**发布（Step 5）** — 平台建议 + 最佳时机 + 配套素材方向

---

### Phase D：项目规划头脑风暴

**背景扫描（Step 1）** — 明确：
- 目标用户是谁 / 要解决什么痛点
- 现有解决方案的缺陷
- 己方差异化优势

**方向发散（Step 2）** — 从以下 4 个视角各提 3-5 个方向：
- 用户视角：他们最想要什么
- 竞争视角：市场空白在哪
- 技术视角：什么能力可以利用
- 商业视角：如何可持续

**路径对比（Step 3）** — 对每条路径评估：时间到首个里程碑 / 资源需求 / 成功标志

**聚焦（Step 4）** — 推荐最值得先验证的 1 个方向，给出 4 周内的验证计划

**死亡预演（Step 5）** — 假设 6 个月后项目失败，最可能的 3 个死因是什么？如何提前规避？

---

## Using Subagents for Devil's Advocate

在 Phase A Step 4 和 Phase D Step 5，使用子 Agent 进行独立反驳：

```
用独立视角检验 [方向/计划]，你是一位严格的投资人/架构师/产品专家。
你的任务：找出最有力的反对理由，不要被上文的分析影响。
给出 3 个最有力的反对意见 + 每个的最佳应对策略。
```

这样可以避免 confirmation bias，得到更可信的分析。

## Output Format

每次头脑风暴结束后，用以下格式收尾：

```
## 决策摘要
- 推荐方向：[Top 1]
- 核心理由：[一句话]
- 最大风险：[一句话]
- 下一步行动：[具体可执行的第一步，最晚本周内可以开始]
```

## Common Mistakes

| 错误 | 修正 |
|------|------|
| 发散阶段就开始评判，砍掉"荒谬"想法 | 先列完 20 个，再收敛 |
| 直接推荐 Top 1，跳过收敛矩阵 | 矩阵是必要的，帮用户看清全局 |
| 技术选型只对比技术指标，忽略团队因素 | 学习曲线和维护成本同等重要 |
| 内容头脑风暴只给选题，没有发布策略 | 好内容也需要渠道策略 |
| 没有 Devil's Advocate 步骤 | 这是避免 confirmation bias 的关键 |
