# mufeng-virtual-team

一个 Agent Skill：让 AI 依次扮演 **CTO、产品经理、普通用户** 三个角色，评审同一个产品功能是否值得做，最后给出综合裁决（做 / 缩减后做 / 不做）和最小实现方案。

适合独立开发者——一个人做产品最缺的不是执行力，而是视角。这个 skill 让你在写第一行代码之前，先借三双眼睛看问题：

- **AI CTO**：挑技术风险——架构耦合、数据迁移、同步冲突、维护成本，以及有没有更小的实现方案
- **AI 产品经理**：挑战需求本身——区分「用户需求」和「开发者想做的功能」，划定 MVP 边界
- **AI 用户**：模拟首次使用——哪里看不懂、哪一步会放弃、要不要先学习才能用

## 安装

```bash
git clone https://github.com/<your-username>/mufeng-virtual-team.git
cp -r mufeng-virtual-team ~/.claude/skills/
```

重启 Claude Code（或新开会话）后生效。

skill 采用通用的 [Agent Skills 规范](https://agentskills.io/specification)，也可用于其他支持该规范的 AI 编程工具，放入对应工具的 skills 目录即可。

## 使用

在 Claude Code 里：

```
/mufeng-virtual-team 我想给 App 加一套智能分类功能，涉及数据模型调整、本地存储与同步、新的界面交互……
```

或直接用自然语言触发：

- 「用虚拟团队评审一下这个功能」
- 「这个功能值不值得做？」
- 「帮我评估这个功能」

背景信息给得越具体（目标用户、使用频率、现有数据模型、项目阶段），评审质量越高。信息不足时 skill 会先追问，不会凭空评审。

## 输出

一份结构化评审报告（markdown），包含：

1. **CTO 视角**：风险清单（按严重度排序）+ 分层实现建议
2. **产品经理视角**：功能真正解决的用户问题 + MVP 边界
3. **用户视角**：第一人称的首次使用反馈
4. **综合裁决**：三方结论对照、冲突点取舍、最终建议（做 / 缩减后做 / 不做）、下一步验证方式

报告默认保存到当前目录（可在对话中指定其他位置，或修改 `SKILL.md` 的「收尾」一节自定义默认路径）。

## 边界

- AI 用户不是真实用户，只能模拟常见行为——**AI 用来暴露问题，真实用户用来验证问题**
- skill 只做分析和建议，不会替你写实现代码
- 背景描述不准确，结论就会偏：输入质量决定评审质量
